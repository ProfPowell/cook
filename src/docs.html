Quick Start

Cook requires Node.js 18+. Clone the repo, install dependencies, and build.

npm install npm run build

For local development with live reload:

npm run dev

Cook reads HTML from src/, processes it through a plugin pipeline, and writes static output to dist/. Every .html file in src/ becomes a clean-URL directory in dist/.

src/about.html → dist/about/index.html

src/blog/my-post.html → dist/blog/my-post/index.html

src/index.html → dist/index.html

Project Structure

my-site/ ├── config/ │ ├── main.js # Site configuration │ └── data.js # Global data (optional) ├── src/ │ ├── index.html # Homepage │ ├── about.html # Pages become /about/ │ ├── blog/ │ │ └── my-post.md # Markdown content │ ├── assets/ │ │ ├── css/ │ │ └── images/ │ ├── components/ # Reusable HTML components │ ├── includes/ # Shared partials (header, footer) │ └── layouts/ # Markdown page layouts ├── dist/ # Build output (git-ignored) ├── scripts/ # Cook build system └── package.json

Build-Only Directories

Directories listed in buildOnlyPaths are used at build time but never copied to dist/. This prevents stale template files from being served as pages.

export default { srcPath: 'src', distPath: 'dist', buildOnlyPaths: ['components', 'includes', 'layouts'], };

Configuration

All configuration lives in config/main.js. It exports a plain object with sensible defaults — you only need to set what you want to change.

export default { distPath: 'dist', srcPath: 'src', startPath: 'index.html', // Directories used only at build time buildOnlyPaths: ['components', 'includes', 'layouts'], // Sitemap generation sitemap: { url: 'https://www.example.com', excludePaths: [/\/assets/, /^\/404.html/], }, // Auto-detect which components need JS autoComponents: { enabled: true, cssOnly: [ 'layout-center', 'layout-grid', 'layout-card', 'layout-cover', 'layout-stack', 'layout-cluster', 'layout-sidebar', 'brand-mark', ], }, // Pre-render Shadow DOM at build time dsd: { enabled: true, manifest: 'assets/vendor/vanilla-breeze/dsd-manifest.json', }, // Component template location components: { path: 'components', }, // Multi-format output formats: { markdown: true, json: true, feed: { title: 'My Blog', collection: 'blog', limit: 20 }, llmsTxt: true, }, // Markdown processing markdown: { layout: 'post', layoutPath: 'layouts', collections: true, markedOptions: { gfm: true }, }, // Paths to exclude from processing excludePaths: [/dist\/assets\/vendor/], };

Data

Global data lives in config/data.js. Export an object or an async function. Every key becomes available as a template variable in your HTML.

Static Data

export default { siteName: 'My Site', currentYear: '2026', navItems: [ { url: '/', label: 'Home' }, { url: '/about/', label: 'About' }, { url: '/blog/', label: 'Blog' }, ], products: [ { name: 'Widget', price: '$29', url: '/products/widget/' }, { name: 'Gadget', price: '$49', url: '/products/gadget/' }, ], };

Async Data

Export a function for data that needs to be fetched at build time — from a database, API, or file system.

import Database from 'better-sqlite3'; export default async function() { const db = new Database('./content.db'); const posts = db.prepare(` SELECT title, slug, description, date FROM posts WHERE status = 'published' ORDER BY date DESC `).all(); return { siteName: 'My Site', collections: { blog: posts }, }; }

Template Strings

Cook uses ES6 template literal syntax (${...}) for variable interpolation. Any key from data.js is available in your HTML.

<title>${siteName} — My Page</title> <p>&copy; ${currentYear} ${siteName}. All rights reserved.</p> <footer>Built with Cook.</footer>

Template strings resolve last in Cook's pipeline. This means variables work inside includes, components, and repeated elements.

Plugin Order

Understanding the pipeline order matters when debugging. Cook processes each page in this sequence:

1. Includes — data-include partials inserted 2. Repeat — data-repeat elements cloned from data 3. Components — custom element templates expanded 4. Template Strings — ${var} resolved across the full page 5. Auto-Components — JS imports optimized per page 6. DSD — Shadow DOM pre-rendered 7. Minify — HTML/CSS/JS minified

Includes

Shared partials (headers, footers, navigation) use the data-include attribute. The value is a path relative to src/.

<!-- Include a shared header --> <div data-include="/includes/header.html"></div> <main> <h1>Page Content</h1> </main> <!-- Include a shared footer --> <div data-include="/includes/footer.html"></div>

The include file is plain HTML:

<header class="site"> <a href="/"> <strong>${siteName}</strong> </a> <nav aria-label="Main"> <ul> <li><a href="/">Home</a></li> <li><a href="/about/">About</a></li> <li><a href="/blog/">Blog</a></li> </ul> </nav> </header>

Template variables like ${siteName} work inside includes because includes are resolved before template string processing.

Components

Cook supports reusable HTML components stored in src/components/. A component is just an HTML file named after the custom element tag.

Creating a Component

<article class="product-card"> <h3><a href="${url}">${name}</a></h3> <p class="price">${priceDisplay}</p> <p>${description}</p> <p class="features">${features}</p> </article>

Using a Component

Use the component as a custom element. Attributes map to template variables inside the component.

<product-card data-name="Trail Runner Pack" data-price-display="$189" data-description="32L ultralight pack." data-features="Waterproof &middot; 680g" data-url="/products/trail-runner-pack/"> </product-card>

Cook replaces the custom element with the component template, substituting data-* attribute values for the matching ${var} placeholders. The data- prefix is stripped and camelCase is converted: data-price-display becomes ${priceDisplay}.

Data-Driven Repeat

The data-repeat attribute clones an element for each item in a data array. The syntax is data-repeat="arrayName as itemName".

<!-- Repeat a card for each product --> <layout-grid data-layout-min="18rem"> <product-card data-repeat="products as product" data-name="${product.name}" data-price-display="${product.priceDisplay}" data-description="${product.description}" data-url="${product.url}"> </product-card> </layout-grid>

Limiting Results

Use the pipe syntax to limit how many items are rendered:

<!-- Show only the 3 most recent blog posts --> <layout-card data-repeat="collections.blog | limit:3 as post"> <article> <h3><a href="${post.url}">${post.title}</a></h3> <p>${post.description}</p> </article> </layout-card>

Nested Data

Access nested properties with dot notation. Collections from markdown are available at collections.collectionName.

Markdown & Collections

Drop .md files into src/ and Cook converts them to HTML pages using a layout template. Front matter defines metadata.

--- title: Trail Running Basics description: Everything you need to know to start trail running. date: 2026-01-15 collection: blog tags: [running, beginners] --- # Trail Running Basics Trail running combines the cardiovascular benefits of road running with the mental engagement of navigating natural terrain. ## What You Need - **Shoes** with aggressive tread and rock plates - **Hydration** vest or handheld bottle - **Navigation** — know your route

Layouts

Markdown pages are wrapped in a layout from src/layouts/. The default layout is configured in config/main.js.

<!doctype html> <html lang="en"> <head> <meta charset="utf-8"> <title>${title} — ${siteName}</title> <meta name="description" content="${description}"> <link rel="stylesheet" href="/assets/css/styles.css"> </head> <body> <div data-include="/includes/header.html"></div> <main id="main"> <article> ${content} </article> </main> <div data-include="/includes/footer.html"></div> </body> </html>

Collections

When collection: blog is set in front matter, the page is added to collections.blog. Use data-repeat to list collection items on any page.

markdown: { layout: 'post', // default layout in src/layouts/ layoutPath: 'layouts', collections: true, // enable front matter collections markedOptions: { gfm: true }, }

Auto-Component Islands

Cook scans each page for custom element tags and determines which ones need JavaScript. CSS-only layout components load zero JS. Pages that only use layout elements have the VB script bundle removed entirely.

autoComponents: { enabled: true, cssOnly: [ 'layout-center', 'layout-grid', 'layout-card', 'layout-cover', 'layout-stack', 'layout-cluster', 'layout-sidebar', 'brand-mark', ], }

How It Works

Page HTML ↓ Scan for custom elements (tags containing a hyphen) ↓ Check against cssOnly list ↓ ├── ALL elements are CSS-only → remove VB <script> tag (zero JS) └── Some elements need JS → keep VB <script> tag

A page using only <layout-grid> and <layout-card> loads no JavaScript from VB. A page using <data-table> or <audio-player> keeps the bundle.

Declarative Shadow DOM

Cook can pre-render Shadow DOM content at build time using Declarative Shadow DOM (DSD). This eliminates the flash of unstyled content for Shadow DOM components and makes their content visible before JavaScript loads.

dsd: { enabled: true, manifest: 'assets/vendor/vanilla-breeze/dsd-manifest.json', }

Before & After

Without DSD, a Shadow DOM component is invisible until JS runs:

<!-- Invisible until JS loads and creates shadow root --> <icon-wc name="check" set="lucide"></icon-wc>

With DSD enabled, Cook injects the shadow root at build time:

<icon-wc name="check" set="lucide"> <template shadowrootmode="open"> <style> :host { display: inline-flex; width: 1em; height: 1em; } svg { width: 100%; height: 100%; } </style> <slot></slot> </template> </icon-wc>

The DSD Manifest

The manifest maps component tag names to their shadow root HTML. Cook reads this file and injects the content as a <template shadowrootmode="open"> inside each matching element.

{ "icon-wc": "<style>:host{display:inline-flex;...}</style><slot></slot>", "audio-player": "<style>:host{display:block;...}</style><div part=\"player\">...</div>" }

Components that use Light DOM (most interactive VB components) don't need DSD — they already work without JavaScript via native HTML fallbacks.

Fragment Generation

Cook generates content-only fragments for SPA-style navigation. For every index.html in dist/, a sibling _fragment.html is created containing only the <main> element's content.

dist/about/ ├── index.html # Full page (html, head, body) └── _fragment.html # <main> content only

Client-side routers (like html-star) fetch the fragment on navigation and swap it into the page, avoiding a full page reload while still supporting View Transitions.

Click link → fetch /about/_fragment.html → swap into <main> → View Transition plays → URL updated via History API

Multi-Format Output

Cook generates the same content in multiple formats. Each page can be served as HTML, markdown, or JSON from the same URL structure.

formats: { markdown: true, // dist/md/{path}/index.md json: true, // dist/api/{path}.json feed: { // dist/feed.xml (Atom) title: 'My Blog', collection: 'blog', limit: 20, }, llmsTxt: true, // dist/llms.txt }

Output Structure

dist/ ├── blog/ │ └── my-post/ │ ├── index.html # Full HTML page │ └── _fragment.html # Content fragment ├── md/ │ └── blog/ │ └── my-post/ │ └── index.md # Markdown source ├── api/ │ └── blog/ │ └── my-post.json # JSON metadata ├── feed.xml # Atom feed └── llms.txt # LLM-readable index

Content Negotiation

When deployed with the Cloudflare Worker, the same URL serves different formats based on the Accept header:

# HTML (default) curl https://example.com/blog/my-post/ # Markdown curl -H "Accept: text/markdown" https://example.com/blog/my-post/ # JSON curl -H "Accept: application/json" https://example.com/blog/my-post/ # Fragment (for SPA navigation) curl -H "X-Requested-With: htmlstar" https://example.com/blog/my-post/

Dev Server

Cook's dev server uses BrowserSync with live reload. It also includes middleware for content negotiation, so fragment routing and format switching work during development.

npm run dev

The dev server:

  • Watches src/ for changes and rebuilds automatically
  • Serves from dist/ with live reload
  • Routes fragment requests (X-Requested-With: htmlstar) to _fragment.html
  • Serves markdown and JSON formats via Accept headers

Deployment

Cook's output is static files. Deploy dist/ anywhere — Cloudflare Pages, Netlify, Vercel, GitHub Pages, or a plain web server.

Cloudflare Workers (Optional)

For content negotiation at the edge, deploy the included Worker. It routes requests to the correct format based on headers and falls back to static files.

# Build the site npm run build # Deploy the worker + static files npx wrangler deploy

Static Hosting

Without the Worker, everything still works. HTML pages are served normally, and you can link directly to /md/, /api/, and _fragment.html paths.

# Build npm run build # Upload dist/ to your hosting provider # e.g., rsync, scp, or CI/CD pipeline

Build Pipeline Summary

Here is the complete flow from source to output:

CREATE dist/ ↓ COPY src/ → dist/ (skip buildOnlyPaths) ↓ PROCESS MARKDOWN (.md → .html with layouts + collections) ↓ FOR EACH .html FILE: │ ├── 1. Replace includes (data-include → inline HTML) ├── 2. Repeat collections (data-repeat → cloned elements) ├── 3. Replace components (custom elements → templates) ├── 4. Template strings (${var} → resolved values) ├── 5. Auto-components (strip unused JS imports) ├── 6. Declarative Shadow DOM (inject <template shadowrootmode>) ├── 7. Fix external links (add missing http://) ├── 8. Inline assets (data-inline → embedded CSS/JS) ├── 9. Set active links (mark current page nav links) ├── 10. Bundle collection (group CSS/JS for bundling) └── 11. Minify (compress HTML/CSS/JS) ↓ GENERATE FRAGMENTS (_fragment.html for each page) ↓ GENERATE FORMATS (markdown, JSON, feed, llms.txt) ↓ GENERATE SITEMAP (sitemap.xml) ↓ BUILD BUNDLES (combined CSS/JS files) ↓ DONE → dist/ ready to deploy
